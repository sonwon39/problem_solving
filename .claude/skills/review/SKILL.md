# /review — 코드 리뷰 & 디버깅 스킬

사용자가 작성한 백준 풀이 코드를 읽고 문제점을 찾습니다.

## 입력 파싱

- `/review cpp/1149.cpp` — 파일 경로 주어진 경우 Read로 불러옴
- `/review 1149` — 문제 번호만 → `cpp/1149.*` 또는 `python/1149.*` 를 Glob으로 찾음
- `/review` + 코드 블록 — 다음 메시지 또는 현재 메시지의 코드를 대상으로

추가 힌트가 있으면 파싱:
- "WA 받았어" → 반례 찾기 모드
- "TLE 났어" → 복잡도 분석 모드
- "RE 났어" → 런타임 에러(배열 범위, 0 나눗셈, 스택 오버플로우) 모드

## 리뷰 체크리스트

코드를 받으면 아래 순서로 분석합니다.

### 1. 제약 조건 대조
- 문제의 N, M, 시간/메모리 제한을 확인 (필요하면 `https://www.acmicpc.net/problem/{번호}` WebFetch)
- 현재 코드의 복잡도가 제약 조건 안에 들어오는지 먼저 판정
- 이 단계에서 이미 TLE가 예상되면 바로 알려주고 접근법 재설계 제안

### 2. 자료형/오버플로우
- `int` vs `long long` (C++) — N*M이 2·10^9를 넘을 수 있는지
- Python은 overflow 걱정 없지만 출력 속도 문제(`print` vs `sys.stdout.write`)
- 입력 속도 (`cin` / `scanf` / `sys.stdin.readline`)

### 3. 경계 조건
- N=1, N=최대
- 빈 입력, 음수, 0, 중복
- 배열 인덱스 off-by-one
- 그래프: 자기 자신으로의 간선, 사이클, 연결되지 않은 노드

### 4. 로직 오류
- 점화식이 맞는지 (DP)
- 방문 처리 위치 (BFS/DFS)
- 초기값 (INF, -INF, 0)
- 출력 형식 (개행, 공백, 소수점 자릿수)

### 5. 반례 제안
위에서 의심 가는 부분을 찾으면, **작은 반례(N ≤ 5)** 를 직접 만들어서
"이 입력에서 코드가 어떻게 동작하는지 추적해보세요"로 전달합니다.

## 답변 규칙

1. **정답을 바로 수정해주지 않습니다.** 문제점을 지적하고, 어떻게 고쳐야 할지 *방향*을 주세요. 사용자가 "고쳐줘"라고 명시적으로 요청하면 그때 수정.

2. **가장 큰 문제 하나부터.** 여러 문제가 있어도 한 번에 하나씩. 사용자가 먼저 그것을 고치고 다시 물어보게 유도합니다.

3. **복잡도는 항상 먼저.** 로직을 보기 전에 "O(N²) 인데 N=10^5이면 10^10 — 이건 TLE입니다" 같은 판정부터.

4. **코드 라인을 인용할 때**는 `file.cpp:42` 형태로.

5. 한국어로 답변.

## 금지 사항

- 사용자가 작성한 코드를 완전히 다시 쓰고 "이게 정답"이라고 주기 (교체가 아니라 교정)
- 스타일 지적만 하고 본질적인 버그를 놓치기
- 제약 조건 확인 없이 "이 코드는 맞는 것 같은데요" 답하기
